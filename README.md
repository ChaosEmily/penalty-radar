# 裁罰案件追蹤器 (Penalty Radar)

專為金融機構風險管理部門設計的 AI 輔助工具。自動監測金管會裁罰案件 RSS，利用 Claude AI 分析違規事實與風險啟示，並自動寄送警示信給風管人員。

## 核心功能與業務價值

本工具不只是技術監測，更是為風管部門設計的**合規閉環管理解決方案**。

### 1. 監測與識別 (Monitoring & Identification)

- **自動監測**：追蹤金管會裁罰案件 RSS，即時掌握同業裁罰動態。
- **AI 風險分析**：使用 Claude Sonnet 4.6 語義理解技術，自動提取受罰對象、罰鍰金額、違反法條、風險啟示。
- **同業篩選器**：依據 `company_type` 自動判斷案件關聯度，精準標註「同業警示」，避免警示疲勞。

### 2. 行動與改善 (Action & Improvement)

- **自我檢核清單 (Action-Oriented)**：AI 根據裁罰事由產出具體應檢視項目，可直接作為內部稽核或業務單位（BU）的檢核底稿。
- **自動產製草稿**：產生 HTML 郵件通知，含 `.eml` 草稿附件與建議主旨，加速內部通報流程。
- **重複違規偵測**：自動比對歷史紀錄，識別累犯機構，協助評估該領域的系統性風險。

### 3. 治理與分析 (Governance & Analysis)

- **KRI 風險指標統計**：產生年度/季度統計報告，監測「五大違規態樣」、「年度罰鍰總支出」等關鍵指標，支援董事會報告。
- **執行紀錄 (Audit Ready)**：`run_history.jsonl` 詳實記錄每次執行狀態，可作為外部風險監測受控的**稽核佐證（Evidence）**。
- **HTML 歷史報告**：按月分類存檔，確保風險監控軌跡不中斷。

## 監測範圍

預設監測以下金管會 RSS 來源：

| 來源名稱 | 公告類型 | 說明 |
| -------- | -------- | ---- |
| 金管會裁罰案件 | 裁罰 | 銀行、保險、證期各業別的裁罰處分公告 |

如需追蹤其他來源，可在 `config.json` 的 `rss_sources` 陣列中新增項目。

## 安裝步驟

1. **複製專案**：

   ```bash
   git clone https://github.com/ChaosEmily/penalty-radar.git
   cd penalty-radar
   ```

2. **建立虛擬環境**：

   本工具建議使用 **Python 3.8+** 以上版本。

   ```bash
   python -m venv .venv
   source .venv/bin/activate  # Windows 請用 .venv\Scripts\activate
   ```

3. **安裝依賴**：

   ```bash
   pip install -r requirements.txt
   ```

4. **初始化設定**：

   - 執行腳本，首次執行會自動產生 `config.json`：

     ```bash
     python setup.py
     ```

   - 開啟 `config.json` 填寫 Gmail 帳號、應用程式密碼與 Anthropic API 金鑰後，再次執行 `setup.py`。
   - 該腳本會自動在 **Windows 工作排程器** 中建立定時任務（預設每 60 分鐘檢查一次）。

## 如何使用

### 手動執行

```bash
python scraper.py
```

### 設定說明 (config.json)

| 欄位 | 說明 |
| ---- | ---- |
| `gmail_user` | 發信用 Gmail 帳號 |
| `gmail_app_password` | Gmail 應用程式密碼（非登入密碼，需至 Google 帳戶設定產生） |
| `recipient_email` | 風管部門收件信箱 |
| `email_strategy` | `priority`（同業即時+非同業延遲，預設）、`single_emails` 或 `digest` |
| `digest_hold_hours` | `priority` 模式下，非同業案件累積多少小時後彙整寄出（預設 24） |
| `scrape_interval_minutes` | 排程執行間隔（分鐘），預設 60 |
| `anthropic_api_key` | Claude API 金鑰 |
| `company_type` | 本公司業別（如 `銀行`、`證券`、`保險`），用於判定同業警示 |

### 通知頻率說明

`scrape_interval_minutes`（排程檢查頻率）與風管人員實際收到信的頻率**不相同**：

- **排程檢查**：系統每隔指定分鐘（預設 60 分鐘）檢查一次金管會 RSS。若無新裁罰則靜默結束。
- **Email 通知**：僅在發現新裁罰時才觸發。實際收信頻率取決於裁罰公告的發布節奏。

以 `priority` 模式為例：

| 情境 | 排程行為 | 風管人員收信 |
| ---- | -------- | ------------ |
| 無新公告 | 定期檢查，靜默結束 | 不收信 |
| 偵測到「同業」裁罰 | 立即處理並發信 | 收到 1 封【同業裁罰警示】 |
| 偵測到「非同業」裁罰 | 加入彙整佇列 | 暫不收信 |
| 非同業案件累積滿 24 小時 | 下次排程檢查時彙整 | 收到 1 封【非同業裁罰彙整報告】 |

### 信件產出規則

#### 信件標題

系統根據案件屬性與寄送模式自動產生標題：

| 案件類型 | 標題格式 | 範例 |
| -------- | -------- | ---- |
| 同業案件 | `🚨【同業裁罰警示】{受罰機構} - {違反法令縮敘}` | 🚨【同業裁罰警示】某某銀行 - 違反洗錢防制法處分案 |
| 一般案件 | `⚖️【裁罰案件追蹤】{受罰機構} - {違反法令縮敘}` | ⚖️【裁罰案件追蹤】某某保險 - 廣告控管缺失處分案 |
| 彙整報告 | `📋【裁罰案件定期彙整】共有 N 則新裁罰資訊` | 📋【裁罰案件定期彙整】共有 5 則新裁罰資訊 |

#### 寄送模式說明

- **priority (推薦)**：同業案件（命中 `company_type`）會第一時間單獨寄出明細；非同業案件則先暫存，每隔 `digest_hold_hours`（預設 24 小時）彙整一次寄出，平衡即時性與信箱整潔。
- **single_emails**：不分業別，每則裁罰案件均獨立寄出一封信。
- **digest**：同一次排程偵測到的多則案件合併為一封寄出。

#### 信件內容結構

每封通知信包含以下區塊：

| 區塊 | 說明 |
| ---- | ---- |
| **裁罰快報** | 標記受罰對象、罰鍰金額、與本公司關聯度（同業/非同業） |
| **AI 風險要點** | 提取具體违規事實與違反法條 |
| **風管建議** | AI 分析該案件對本公司的風險啟示 |
| **自我檢核清單** | 針對該違規事由，提供的具體內部檢查建議項目 |
| **內部通知草稿** | 可直接轉寄給相關業務單位（如分行、後勤）的通知範本 |

## 檔案結構

- `scraper.py`: 核心抓取、AI 分析與發信邏輯。
- `stats.py`: 裁罰趨勢統計報告產生器（`python stats.py` 或 `python stats.py 2025`）。
- `setup.py`: 初始化設定與排程註冊。
- `config.json`: 系統設定（執行時自動產生，不入版控）。
- `processed_penalties.json`: 已處理的裁罰案件紀錄（不入版控）。
- `penalty_history.json`: 機構裁罰歷史（供重複違規偵測與統計，不入版控）。
- `pending_digest.json`: 非同業案件彙整佇列（`priority` 模式使用，不入版控）。
- `reports/`: 自動產生的 HTML 月報、年度統計與 KRI 報表。

## 支援平台

- **Windows 10/11**：完整支援，含 Windows 工作排程器自動註冊。
- **Linux / macOS**：
    1. `scraper.py` 可正常執行。
    2. 自動化排程需手動設定 Crontab：
       - 執行 `crontab -e`
       - 加入以下設定（範例為每小時執行）：

         ```bash
         0 * * * * cd /path/to/penalty-radar && /path/to/venv/bin/python scraper.py >> cron.log 2>&1
         ```

## 資安與隱私保護

- **本地端優先**：所有設定（如明文 API Key、發信帳密）僅儲存於本地 `config.json`，不入版控。
- **數據最小化**：系統僅將「金管會公開公告內容」發送至 Claude 進行語義分析，不涉及公司內部策略或客戶個資上傳。
- **透明監控**：所有 AI 交談紀錄與執行軌跡均記錄於本地 `run_history.jsonl` 供內部資安查核。

## 資料保留

- HTML 月報與年度報告存放於 `reports/` 目錄，系統啟動時會自動檢查超過 **5 年** 的報告並提醒歸檔。
- `run_history.jsonl` 與 `penalty_history.json` 建議定期備份，作為長期合規監測與風險分析的原始資料。

本工具產出之分析與草稿僅供內部參考，不代表正式法律意見。所有裁罰案件請以主管機關公告及裁罰書全文為準。
